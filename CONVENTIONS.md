# Chart conventions

How charts in this repository are built. Follow these when adding a chart or changing an existing
one, so the charts stay consistent with each other.

Most of the rules below exist because something broke. Where that is the case the reason is stated —
do not "simplify" one away without re-reading it.

`charts/outline` and `charts/request-manager` are the reference implementations.

---

## Layout

```text
charts/<name>/
  Chart.yaml
  values.yaml
  README.md              generated by helm-docs, never hand-edited
  README.md.gotmpl       only when the chart needs an Upgrading section
  ci/                    one file per install scenario, see Testing
  templates/
    _helpers.tpl
    configmap.yaml       non-secret env
    secret.yaml          sensitive env
    NOTES.txt
    tests/test-connection.yaml
```

A chart with more than one workload puts each in its own folder — `templates/server/`,
`templates/worker/`, `templates/beat/` — with shared resources at the `templates/` root. Resources
that front a single workload (Service, Ingress, HPA, PDB) live in that workload's folder.
`$.Template.BasePath` is the chart's templates root, not the current file's directory, so
checksum-annotation includes keep working from subfolders.

## Configuration

Split application settings into two values maps:

- `config:` renders into a **ConfigMap**
- `secrets:` renders into a **Secret**

Keys are the **literal environment variable names the upstream app reads**, taken from its
`.env.sample`. Do not invent a nicer naming scheme and translate it in a helper — a reader should be
able to grep the app's source for a key and find it. Link the upstream `.env.sample` from the values
comment.

Both are consumed with `envFrom`. Empty values are dropped, so blanking a chart default removes the
variable rather than setting it to `""` — for most config libraries those are different things.
Values are rendered through `tpl`, so they can reference `.Release.Name`.

### Derived values

Anything the chart can compute, the chart computes — connection hosts, ports, database names, and
**credentials**. A password must never appear in two values a user has to keep in sync.

```gotemplate
{{- if and .Values.postgres.enabled .Values.postgres.auth.password -}}
{{- $_ := set $computed "DATABASE_PASSWORD" .Values.postgres.auth.password -}}
{{- end -}}
```

User values win over computed ones: `merge $user $computed`.

### existingSecret

`existingSecret` **replaces the user's `secrets` map, but not the chart's derived values**. The
chart's own Secret always renders and keeps what the chart computed; the external Secret is appended
as a second `secretRef` in `envFrom`.

Gating the whole Secret behind `existingSecret` is wrong: it takes the derived database password with
it, so a bundled-database install then needs that password written into the external Secret _and_
into `postgres.auth.password`, and they break silently when they drift.

Anything reading a single key (a CronJob using `secretKeyRef`) must resolve the name through a helper
so it follows `existingSecret`.

## values.yaml naming

| Key                                 | Meaning                                           |
| ----------------------------------- | ------------------------------------------------- |
| `config`, `secrets`                 | application settings, see above                   |
| `existingSecret`                    | external Secret for the sensitive settings        |
| `extraEnv`                          | env entries appended to the chart's own, verbatim |
| `extraEnvFrom`                      | envFrom sources appended to the chart's own       |
| `extraVolumes`, `extraVolumeMounts` | appended to the chart's own                       |
| `initContainers`                    | appended to the chart's own                       |

`extra*` means "added to what the chart already manages". Anything a user should routinely change
belongs in `config`/`secrets` instead — Kubernetes gives `env` precedence over `envFrom`, so entries
in `extraEnv` silently override both the ConfigMap and the Secret.

Probes live in `values.yaml` (`livenessProbe`, `readinessProbe`, `startupProbe`) and are rendered
under `{{- with }}`, so setting one to `null` disables it.

## Rules that are easy to get wrong

**Sub-chart hostnames use `.Release.Name`, never `fullname`.** The CloudPirates sub-charts name
themselves `<release>-postgres`. `<chart>.fullname` only equals the release name when the release
name happens to contain the chart name — which is exactly what CI generates, so the mistake passes
CI and breaks every real install.

**Selectors must be scoped by `app.kubernetes.io/component`.** `<chart>.labels` is applied to every
pod the chart creates, including CronJob runners and the `helm test` pod. A Service or PDB selecting
on `selectorLabels` alone will match them. Put the component label on **pod templates only** —
`spec.selector.matchLabels` on a Deployment is immutable, so adding it there breaks upgrades of
existing releases.

**Security contexts are merged as dicts, never appended as YAML.**

```gotemplate
{{- toYaml (mustMergeOverwrite $defaults (deepCopy (default dict .Values.securityContext))) }}
```

Appending produces duplicate YAML keys, and a nested override such as `capabilities.add` replaces
the default `capabilities.drop: [ALL]` instead of extending it — silently unhardening the container.

**`pathType` is required** on `networking.k8s.io/v1` Ingress. Default it: `{{ .pathType | default "Prefix" }}`.

**Memory `requests` equals `limits`.** Memory is incompressible, so a lower request buys no burst
headroom — it only understates the pod to the scheduler and makes it an earlier eviction candidate.
CPU is the opposite: keep its request below the limit.

**Liveness must not depend on external services.** If the app's health endpoint checks its database,
liveness on that endpoint restarts every replica during a database blip, which cannot possibly fix
it. Readiness may use it — removing a pod from the Service is the correct response. Use `tcpSocket`
for liveness when the app exposes no process-only endpoint, and comment why.

**`terminationGracePeriodSeconds` must exceed the app's own shutdown timeout.** Look it up in the
app's source; the Kubernetes default of 30s kills many apps mid-drain.

**`kubeVersion` matches the highest GA version the chart renders.** `autoscaling/v2` means
`>=1.23.0-0`. A floor that is too low passes the gate and then fails at apply time.

**`runAsUser` must be above 10000** (Checkov CKV_K8S_40). Prefer 65532/65534 depending on the image.

## Testing

Every chart ships these `ci/` scenarios. `ct install` runs one install per file, so each is a real
deployment on kind and each must be self-sufficient.

| File                             | Exists to catch                                                                          |
| -------------------------------- | ---------------------------------------------------------------------------------------- |
| `default-values.yaml`            | the default path works end-to-end                                                        |
| `ingress-values.yaml`            | ingress on, **`pathType` omitted** so the chart default is what the API server validates |
| `security-overrides-values.yaml` | a nested `capabilities.add` that must not drop the default `drop: [ALL]`                 |
| `fullname-override-values.yaml`  | `fullnameOverride` set, so `fullname` diverges from the release name                     |

The last one matters more than it looks: `ct` always generates a release name containing the chart
name, so it structurally cannot catch a `fullname`-vs-release-name mistake. `fullnameOverride`
produces the same divergence by another route.

**`helm template` proves nothing about whether the app starts.** Before opening a PR, check what
settings the upstream app reads with no default (`config("X")` with no fallback, `process.env.X`
dereferenced directly) and make sure the `ci/` files supply them — otherwise `ct install` is the
first thing to find out, one missing variable per run.

## Documentation

`README.md` is generated — run `./scripts/helm-docs.sh`, never edit it. Document values with `# --`
comments; they become the README table. Add a `README.md.gotmpl` only when the chart needs prose the
generator does not produce, such as an Upgrading section for a breaking release.

Keep comments to one line where possible. State the non-obvious fact and stop; do not spell out the
consequences that follow from it. Keep the "why" only where the fact alone does not warn the reader —
`minAvailable: 1` blocking node drains at a single replica, or a health endpoint that queries the
database.

## Working on a chart

Verify claims about the upstream application by reading its source. Whether an endpoint touches the
database, what a shutdown timeout is, which settings are required — these decide chart behaviour and
are cheap to check and expensive to guess wrong.

When refactoring, render before and after and diff them. A change intended to be cosmetic should
produce an identical diff apart from key ordering.
